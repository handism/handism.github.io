## はじめに

最近、プライバシー保護の観点から「すべてのデータをクラウドに送信せず、デバイス内で完結させるエッジAI（ローカルAI）」に関心を持っている。

勉強がてら、Androidスマホの内部ストレージにある複数のマークダウンファイル（日記やメモ）を読み込んで、端末内で質問に答えてくれるAIノートアプリ「`mini-brain`」を開発している。

しかし、実際に実機で動かしてみると、回答の精度が上がらず、実用には程遠い状態だった。そもそもエッジ向けの超軽量モデル（Gemma 2Bなど）は、クラウドの巨大モデル（GPT-4やClaude）と比べて圧倒的に脳みそが小さい。

試行錯誤を繰り返しながら、ローカルRAGの精度向上と、特に大苦戦した「去年の冬何してたっけ？」のような**「時間・期間クエリ」**への対策を施した結果、実用レベルまで回答精度を引き上げることができた。今回はそのチューニングの裏側をすべて書き残しておく。

---

## システム構成とモデルスペック

今回のアプリにおける、想定モデルと開発環境のスペックだ。

| 項目 | スペック / 採用技術 | 役割・制約 |
| :--- | :--- | :--- |
| **ベースOS** | Android | 実機での動作確認を重視 |
| **オンデバイスLLM** | Gemma 2B (LiteRT-LM) | プライバシー優先、単一スレッドで動作（遅い） |
| **埋め込みモデル** | USE (Universal Sentence Encoder) Multilingual | ベクトル検索用。コサイン類似度が低めに出る特性あり |
| **検索エンジニアリング** | SQLite FTS4 + ベクトルDB | キーワード検索（BM25）と意味検索（Vector）のハイブリッド |
| **エージェント手法** | Search First フロー + ReActフォールバック | 処理の高速化と、回答不能時の探索を両立 |

## 1. なぜ「LLMにお任せするエージェント（ReAct）」は失敗したのか？

開発の初期段階では、Claude CodeやCursorエージェントのように、LLMに「どのキーワードで検索（grep）するか」を考えさせ、繰り返しファイルを探索させる**ReActループ**を採用していた。

しかし、これがGemma 2Bでは全く機能しなかった。

<div class="is-style-big_icon_bad">
Gemma 2Bのような軽量モデルは、自律的に「次に何を検索すべきか」を推論する能力が弱い。また、エージェントのループを数往復させるだけでも（単一スレッド動作のため）数十秒のレイテンシが発生し、実用的な速度ではなくなってしまった。
</div>

この失敗から、**「軽量LLMをエージェントとして働かせるのは諦め、検索処理はルールベースのパイプラインで解決した上で、LLMには最後の回答生成だけを任せる（Search First構造）」**という方針へ舵を切った。

## 2. 検索精度を底上げするRAGの3大チューニング

検索をルールベースのパイプラインに移行したものの、デフォルト設定では関連ファイルがうまくヒットしなかった。そこで行ったのが以下の3つのパラメータ調整だ。

### ① 類似度しきい値の緩和（0.45 → 0.30）
使用していた埋め込みモデル（USE Multilingual）は、他のモデルに比べてコサイン類似度のスコアが低めに出る傾向があった。初期設定のしきい値 `0.45` では、関連度の高いチャンクがごっそりフィルタリングで落とされていた。
しきい値を `0.30` まで下げることで、検索漏れ（Recall）が大幅に改善された。

### ② チャンクのオーバーラップを増やす（50文字 → 150〜200文字）
1チャンク800文字に対して、重複部分（オーバーラップ）が50文字（約6%）だと、文章の途中でブツ切りになった際に文脈が失われてしまっていた。これを全体の約20%に相当する150〜200文字に増やすことで、文脈の維持が安定した。

### ③ BM25 × Vector × メタデータの「ハイブリッド並行検索」
ベクトル検索（意味的類似度）だけでは、固有名詞や特定の表記の検索に弱い。そこで、SQLiteのFTS4を利用したキーワード検索（BM25）、ファイル名やタグを見るメタデータ検索、そしてベクトル検索の3本を並行で走らせ、最後にReciprocal Rank Fusion (RRF) で融合する仕組みにした。
また、日本語の1文字検索にも対応できるよう、SQLite用にユニグラム（1文字トークナイズ）を追加した。

## 3. 最大の難所：「去年の冬何してた？」という時間クエリの壁

RAGの検索を強化したことで、「サウナしきじについて教えて」といったトピック検索は完璧になった。しかし、日記アプリとして致命的な問題が発生した。

> ユーザー：「今年の正月って何してたっけ？」
> AI：「知識ベースには今年の正月に関する情報が含まれていません」

実際には、`2026-01-01.md` などの日記が存在するにもかかわらず、AIがそれを引き当てられない。原因は、**「今年の正月」や「去年の冬」といった曖昧な時間表現を、2BクラスのローカルLLMがデータベース検索用の日付範囲（`2026-01-01`〜`2026-01-07`）に翻訳できなかったこと**だ。

この「時間クエリ」問題を解決するために、以下の3ステップの対策を実装した。

### ステップA：ルールベースの「DateResolver」の実装
LLMに日付の推論を任せるのをやめ、クライアントサイド（Kotlin）でクエリ内の時間表現を正規表現で解釈する `DateResolver` を導入した。

```kotlin
// 「昨日」「正月」「去年の冬」などを検出し、具体的な日付/日付範囲に変換する
object DateResolver {
    private val MONTH_DAY_RE = Regex("""(\d{1,2})月(\d{1,2})日""")
    private val MONTH_ONLY_RE = Regex("""(\d{1,2})月""")

    fun resolveToDateStrings(question: String): List<String> {
        val today = LocalDate.now()
        val isPastYear = question.contains("去年") || question.contains("昨年")
        
        return when {
            question.contains("正月") -> {
                val year = if (isPastYear) today.year - 1 else today.year
                (1..7).map { LocalDate.of(year, 1, it).toString() } // 1/1〜1/7の範囲
            }
            question.contains("年末") -> {
                // 上半期（6月以前）に「年末」と言ったら去年の年末を指す可能性が高いという推定ルール
                val year = if (isPastYear || today.monthValue <= 6) today.year - 1 else today.year
                (25..31).map { LocalDate.of(year, 12, it).toString() }
            }
            // 「1月1日」や「2月（月全体）」のパターンの解析も正規表現でフォールバック
            else -> resolveByNumericPattern(question, today)
        }
    }
}
```

この「年の指定がない場合は、未来の月なら去年、過去の月なら今年と推定する（例：6月時点で『8月何した？』なら去年の8月）」といった人間らしい時間推論のルール化により、検索対象の日付を確実に特定できるようになった。

### ステップB：検索結果への「[日付: YYYY-MM-DD]」プレフィックス強制付与とピン留め
検索で引き当てた日記のテキストをLLMに渡す際、ただテキストを渡すだけでは、LLMが「その日記がいつのものか」を認識できない。
そこで、抽出したメタデータから `[日付: YYYY-MM-DD]` というプレフィックスを各スニペットの先頭に明示的に埋め込むようにした。

```
[日付: 2026-01-02] 初回訪問日: 2022/01/02。サウナしきじに到着。やっぱり水風呂が神すぎる...
```

さらに、ベクトル検索のReranker（順位再評価）によって、期間対象の日記が下位に埋もれてしまうのを防ぐため、`DateResolver` で解決された日付範囲に合致するファイルを、**Rerankerの評価をバイパスしてコンテキストの最優先位置（先頭5件）に強制的にピン留めする（Pinning）** 処理を検索パイプラインの最後に追加した。

### ステップC：回答LLM向けプロンプトへの期間情報注入
最後に、回答を生成するプロンプトに対しても、以下のように解決された期間情報を明示的に与え、LLMの目線を合わせた。

<div class="is-style-big_icon_point">
「質問は 2024-12-01 〜 2025-02-28（去年の冬）の期間に関するものです。各スニペット先頭の <code>[日付: YYYY-MM-DD]</code> をこの期間と照合し、最優先で根拠にしてください。」
</div>

この誘導を行うことで、Gemma 2Bは「去年の冬の日記スニペット」と「ユーザーからの質問」を頭の中で結びつけることができるようになり、「去年の冬は、〇〇サウナに行っていましたね」と的確に答えられるようになった。

## 4. 改善の結果と得られた知見

これらのチューニングにより、実機テストでの回答率は劇的に改善した。

### よかった点（Good）
- **「いつ行った？」に即答可能**: 日付情報を含むクエリの精度（Recall）が大幅に向上した。
- **回答の高速化（レイテンシ削減）**: 時間クエリと判断した時点で、重たいLLMによる「回答可能かどうかの判定（CoverageCheck）」をスキップして即答フェーズに入る短絡（Short-Circuit）処理を入れたため、数秒の処理時間削減に成功した。
- **SQLite FTSの最適化**: データのインデックス更新処理にトランザクションを導入し、インデックス作成自体のパフォーマンスも劇的に向上した。

### 惜しい点（Bad）
- **ルールベースの漏れリスク**: 「一昨々日（さきおととい）」のような、まだルールに定義されていない極端な相対表現には対応できない。ここはいずれ軽量なローカル parser モデル等に切り替えるか、定義を拡充する必要がある。

---

## まとめ

軽量なオンデバイスモデルを使ってRAGを構築する場合、**「LLMに頼る部分」と「プログラム（ルール）で解決する部分」をいかに明確に切り分けられるか**がすべてだと感じた。

特に日付や時間のような構造化しやすいデータは、LLMの貧弱な推論力に頼るよりも、正規表現や事前ルールで解釈して検索パイプラインを強固にした方が、圧倒的に速く、そして正確な結果が得られる。

「エッジAIは賢くないから使い物にならない」と諦める前に、検索エンジニアリングとプロンプトの渡し方を見直すだけで、端末完結型の素晴らしいプライベートAIアシスタントを作り上げることができる。皆さんもぜひローカルRAGのチューニングに挑戦してみてほしい。
